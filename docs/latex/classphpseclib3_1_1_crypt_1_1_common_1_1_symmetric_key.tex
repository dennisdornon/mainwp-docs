Treat consecutive "{}packets"{} as if they are a continuous buffer.

Say you have a 32-\/byte plaintext \$plaintext. Using the default behavior, the two following code snippets will yield different outputs\+:

{\ttfamily  echo \$rijndael-\/\texorpdfstring{$>$}{>}encrypt(substr(\$plaintext, 0, 16)); echo \$rijndael-\/\texorpdfstring{$>$}{>}encrypt(substr(\$plaintext, 16, 16)); } {\ttfamily  echo \$rijndael-\/\texorpdfstring{$>$}{>}encrypt(\$plaintext); }

The solution is to enable the continuous buffer. Although this will resolve the above discrepancy, it creates another, as demonstrated with the following\+:

{\ttfamily  \$rijndael-\/\texorpdfstring{$>$}{>}encrypt(substr(\$plaintext, 0, 16)); echo \$rijndael-\/\texorpdfstring{$>$}{>}decrypt(\$rijndael-\/\texorpdfstring{$>$}{>}encrypt(substr(\$plaintext, 16, 16))); } {\ttfamily  echo \$rijndael-\/\texorpdfstring{$>$}{>}decrypt(\$rijndael-\/\texorpdfstring{$>$}{>}encrypt(substr(\$plaintext, 16, 16))); }

With the continuous buffer disabled, these would yield the same output. With it enabled, they yield different outputs. The reason is due to the fact that the initialization vector\textquotesingle{}s change after every encryption / decryption round when the continuous buffer is enabled. When it\textquotesingle{}s disabled, they remain constant.

Put another way, when the continuous buffer is enabled, the state of the \textbackslash{}phpseclib3\textbackslash{}\+Crypt\texorpdfstring{$\ast$}{*}() object changes after each encryption / decryption round, whereas otherwise, it\textquotesingle{}d remain constant. For this reason, it\textquotesingle{}s recommended that continuous buffers not be used. They do offer better security and are, in fact, sometimes required (SSH uses them), however, they are also less intuitive and more likely to cause you problems.
